## 成長戦略の大前提

**営業担当者はいない。プロダクト自体が広がる仕組みを持たなければならない。**

これは「プロダクト・レッド・グロース（PLG）」戦略。
広めてくれる人に頼るのではなく、**ユーザーが自然と「使いたい・シェアしたい・戻りたい」と思う設計をプロダクトに組み込む**ことで成長する。

PLGが成立する3条件：
1. **発見される** — AIに聞いたとき・知人のLINEで見たときに出会う
2. **すぐ価値を感じる** — 開いて2分以内に「役に立った」と思える
3. **また来たくなる** — 結果を保存して続きが使える・変化が通知で届く

## フェーズ別 優先順位（必ずこの順番で作業する）

### Phase 1: UX改善（最優先 — 今すぐ取り組む）

> 200枚のカードがあっても、ユーザーが「自分のことだ」と感じて最初の1枚を使えなければ意味がない。

**✅ 実装済み（これ以上の追加不要）:**
- 全5ツール: TOP3ピン留めセクション
- 全5ツール: 検索バー・カテゴリフィルター
- 全5ツール: ウィザード形式の状況別おすすめカード案内
- index.html: 「売りたい/賃貸/投資」3状況別ワンタップ入口

**🔥 今すぐ取り組む（未実装・最重要）:**

1. **カードの見出し・説明文を「一般ユーザー言語」に書き換える**
   - 現状: 「DSCRシミュレーション」「譲渡所得税簡易試算」
   - 目標: 「銀行が貸してくれる金額の安全率チェック」「家を売った後の税金を概算する」
   - 対象: TOP3カードと各ツール上位10枚の見出し・説明文を平易な言葉に修正
   - 実装方法: HTML内のcard-title・説明文の文言を直接書き換える

2. **各ツールに「初めての方はここから」ランディングバナーを設ける**
   - ツール上部（TOP3の上）に「初めての方は → ○○カードだけ使ってみてください」と1枚だけ強く推薦
   - 例: Tool1なら「まずは簡易査定カードだけ試してみてください（入力3項目・30秒）」
   - 決断麻痺を防ぐ「これだけやってみて」の導線

3. **index.html を「ユーザーの悩みベース」の言語に変える**
   - 現状: ツール名の羅列（AI査定 / 空き家 / 売主買主 / 管理 / 投資）
   - 目標: 「親から家を相続した」「家賃を上げたい」「マンション投資を始めたい」などの悩み起点で誘導
   - 「あなたの状況はどれですか？」 → 状況から自動でおすすめツール・カードを表示

4. **TOP3のカード説明を「こんな人に使ってほしい」形式にする**
   - 例: 「🏠 簡易査定 → 家の売却価格の目安を知りたい人へ。築年数と広さを入力するだけ」
   - 機能名でなく、ユーザーの状況・ベネフィットを先に伝える

### Phase 2: 自然な発見・拡散（営業なし・PLG設計）（Phase 1完了後）

> **大前提: 営業担当者はいない。プロダクト自体に「広がる仕組み」を組み込む。**
> 従来のGoogle SEOはAI Overview・ChatGPTの台頭で急速に陳腐化中。
> 狙うべきは「ユーザーが自発的にシェアする」「AIが自然に引用する」の2軸。

#### 2-A. 計算結果の「シェアしたくなるデザイン」（最優先）

> 人が広めてくれるのを待つのではなく、**結果を見た瞬間に「これ、誰かに見せたい」と思わせる**設計。

1. **LINEシェアボタンを計算結果に組み込む**
   - 計算完了時に「LINEで家族に送る」ボタンを目立つ位置に表示
   - 40〜60代の主要連絡手段はLINE。これが最もコストゼロで拡散できる
   - シェアテキストを自動生成：「うちの物件、簡単査定したら約3,200万円って出た！」＋URL

2. **結果を「画像として保存」できるようにする（中長期）**
   - インスタ・X・LINEに貼れる結果カード画像のダウンロード機能
   - 「診断結果シェア」カルチャーを狙う（診断系コンテンツはSNSで自然に広まる傾向）

3. **「誰かに見せたい数字」が出る設計を意識する**
   - 新機能追加時、結果の数字・グレード・スコアが「驚き・共感・不安」を生むか確認する
   - 例：「あなたの空き家、放置コストは10年で1,200万円」→ 思わずシェアしたくなる

#### 2-B. AEO（AI回答エンジン最適化）— AIに発見してもらう

> **従来SEO（Googleクローラー最適化）は補助手段に格下げ。**
> 主戦場は ChatGPT・Perplexity・Google AI Overviews。
> 「不動産 査定 ツール」と聞かれたとき、AIがこのサービスを紹介する状態を作る。

1. **`llms.txt` をルートに設置**
   - AIクローラー向けの「このサービスの説明書」
   - サービス名・URL・機能一覧・対象ユーザー・使い方を平易な文章で記述
   - Perplexity・ChatGPT SearchがこのサービスURLをクロールしたとき正確に把握できる

2. **各ページにFAQ形式のJSON-LD（schema.org/FAQPage）を追加**
   - 「不動産の査定額はどう計算する？」→ 「このツールでは路線価・築年数・面積から自動試算できます」
   - AIが質問に回答する際にこのページを出典として引用しやすくなる

3. **従来SEOの基本設定は維持（AEOと兼用）**
   - `<title>`, `<meta description>`, `<h1>` の適切な設定は続ける
   - キーワード詰め込み型の旧式SEOへの投資は不要。基本設定のみで十分

#### 2-C. 「また来たくなる」仕掛け（リテンション → 口コミの源泉）

> 口コミは「使い続けているユーザー」から生まれる。一度来て終わりにさせない。

1. **計算結果の保存と比較**
   - 「前回の査定結果と今回を比較」できる機能
   - 「3ヶ月前より100万円下がった」という変化が可視化されると再訪動機になる

2. **「お気に入り機能 TOP3」のパーソナライズ**
   - よく使うカードをブックマークして毎回探す手間をなくす
   - 「あなたがよく使う機能」がトップに出る状態

3. **TERRA REALTYへの自然な相談導線（CTAは押しつけず自然に）**
   - 計算結果の下に「もっと詳しく知りたい場合は…」として相談導線を設置
   - 「無料相談」ではなく「この結果の詳細解説を受け取る」などソフトな表現を使う
   - 営業感をなくし、あくまでユーザーの行動の延長として自然に誘導する

### Phase 3: データ活用・継続利用（Phase 2完了後）

> ユーザーが「また来たい」と思う仕掛けと、蓄積データの活用。

1. **使用履歴・計算履歴の改善**
   - 前回の計算結果を表示する「前回の続き」機能
   - 「あなたの査定履歴」「よく使う機能」の個人化

2. **データ蓄積の仕組み**
   - Supabase DBへの匿名集計データ保存（個人を特定しない集計）
   - 「この地域の平均査定額は〇〇万円」のような統計情報の表示

3. **通知・リマインド**
   - 登録ユーザーへのメール通知（査定結果の変化・市場情報）

## 実装ルール

- **重複チェック必須**: 実装前に必ず `improvement-log.md` と `git log` で既存機能を確認し、重複しない改善を選ぶ
- **構文チェック必須**: コミット前に以下で JS 構文エラーがないことを確認する
  ```bash
  node -e "
  const fs = require('fs');
  const html = fs.readFileSync('[ファイルパス]','utf8');
  const match = html.match(/<script(?![^>]*type=\"application\/ld)[^>]*>([\s\S]*?)<\/script>/g);
  let combined = '';
  if(match) match.forEach(s => { combined += s.replace(/<\/?script[^>]*>/g,'') + '\n'; });
  try { new Function(combined); console.log('JS OK'); } catch(e) { console.error('JS Error:', e.message); }
  "
  ```
- **index.html の構文チェック**: JSON-LD ブロックを除外するフィルタを使うこと（上記の正規表現で対応済み）
- **1機能ずつコミット**: 機能単位でコミットし、push まで完了させてから次へ
- **ユーザーへの質問禁止**: 機能の選択・実装方針はすべて自律判断する
- **git add は個別指定**: `git add -A` や `git add .` は使わない。必ず対象ファイルを明示してステージングする
- **Supabase アクセス**: `supabase-client.js` を通じてのみDBアクセスすること。直接fetchでSupabaseを呼ばない
- **localStorageキー名・データ構造を変更しない**: 未ログインユーザーの既存データが消えるため
- **どこかのサーバーにファイルをアップロードしない**
- **関数名・要素ID重複チェック**: 新規追加前に `grep -o 'function calc[A-Za-z]*' [file] | sort | uniq -d` で重複確認する
