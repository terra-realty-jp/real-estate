## 成長戦略の大前提

**営業担当者はいない。プロダクト自体が広がる仕組みを持たなければならない。**

これは「プロダクト・レッド・グロース（PLG）」戦略。
広めてくれる人に頼るのではなく、**ユーザーが自然と「使いたい・シェアしたい・戻りたい」と思う設計をプロダクトに組み込む**ことで成長する。

PLGが成立する3条件：
1. **発見される** — AIに聞いたとき・知人のLINEで見たときに出会う
2. **すぐ価値を感じる** — 開いて2分以内に「役に立った」と思える
3. **また来たくなる** — 結果を保存して続きが使える・変化が通知で届く

## CLAUDE.md 自己更新ルール

**CLAUDE.mdは実装の進捗に合わせて常に最新状態を保つこと。古いCLAUDE.mdは誤った優先順位で作業させる原因になる。**

### 更新タイミング（このいずれかに当てはまったら更新する）
- フェーズが実質的に進んだとき（Phase 1の主要課題がほぼ完了 → Phase 2へなど）
- 「✅ 実装済み」リストに新たに加えるべき項目が3つ以上たまったとき
- 「現在の達成度」が実態と5%以上ずれてきたと感じたとき
- 「🔥 今すぐ取り組む」リストの全項目が完了したとき
- セッション開始時に「現状の客観的評価」の日付が1週間以上前になっていたとき

### 更新内容（この順番で確認・修正する）
1. `tail -100 improvement-log.md` と `git log --oneline -20` で最新の実装状況を確認
2. 「現状の客観的評価」の**日付・カード枚数・達成度・残課題**を実態に合わせて書き換える
3. 各フェーズの「✅ 実装済み」リストに新規完了項目を追加し、「🔥 今すぐ取り組む」から削除する
4. 「🔥 今すぐ取り組む」に新たな残課題があれば追記する
5. 変更した場合は `git add CLAUDE.md && git commit -m "docs: CLAUDE.md現状評価を更新（YYYY-MM-DD）" && git push origin main`

### 更新禁止事項
- 実装が完了していないのに「✅ 実装済み」と書かない
- 「達成度」を実態より高く書かない（楽観的な自己評価はサービス改善を止める）
- このCLAUDE.md自体の「使命」「品質の基準」「実装ルール」は変更しない（恒久ルール）

## 実装ルール

- **重複チェック必須**: 実装前に必ず `improvement-log.md` と `git log` で既存機能を確認し、重複しない改善を選ぶ
- **構文チェック必須**: コミット前に以下で JS 構文エラーがないことを確認する
  ```bash
  node -e "
  const fs = require('fs');
  const html = fs.readFileSync('[ファイルパス]','utf8');
  const match = html.match(/<script(?![^>]*type=\"application\/ld)[^>]*>([\s\S]*?)<\/script>/g);
  let combined = '';
  if(match) match.forEach(s => { combined += s.replace(/<\/?script[^>]*>/g,'') + '\n'; });
  try { new Function(combined); console.log('JS OK'); } catch(e) { console.error('JS Error:', e.message); }
  "
  ```
- **index.html の構文チェック**: JSON-LD ブロックを除外するフィルタを使うこと（上記の正規表現で対応済み）
- **1機能ずつコミット**: 機能単位でコミットし、push まで完了させてから次へ
- **ユーザーへの質問禁止**: 機能の選択・実装方針はすべて自律判断する
- **git add は個別指定**: `git add -A` や `git add .` は使わない。必ず対象ファイルを明示してステージングする
- **Supabase アクセス**: `supabase-client.js` を通じてのみDBアクセスすること。直接fetchでSupabaseを呼ばない
- **localStorageキー名・データ構造を変更しない**: 未ログインユーザーの既存データが消えるため
- **どこかのサーバーにファイルをアップロードしない**
- **関数名・要素ID重複チェック**: 新規追加前に `grep -o 'function calc[A-Za-z]*' [file] | sort | uniq -d` で重複確認する

## 品質の基準

「不動産のことをよく知らない40〜60代の方が、スマートフォンでツールを開いて、**誰にも聞かずに2分以内に目的の操作にたどり着ける**」状態を目指す。

### 合格ライン（この基準を満たすまで次のフェーズに進まない）

**オンボーディング:**
- ページを開いた瞬間に「自分に関係ある」とわかる見出し・言葉が使われている
- 「初めての方はここから」という1枚の推薦カードが目立つ位置にある
- ウィザードで状況を選ぶと、自分に合ったカードが強調表示される

**カードの説明:**
- カードのタイトルを見ただけで、専門知識ゼロの人が「何ができるか」を理解できる
- 入力項目の横に「例: 築15年の場合は15と入力」などの参考値が示されている
- 結果の数字の意味（単位・良い/悪いの判断基準）が結果欄に書かれている

**操作・ナビゲーション:**
- 目標のカードが2タップ以内で見つかる（検索・ピン留め・ウィザードで実現）
- ボタンのラベルを見ただけで何ができるかわかる
- 専門用語には必ず平易な説明が付いている
- スマホで操作しやすいボタンサイズ（最低44px）・フォントサイズ（最低13px）

### 実装時の言語チェックリスト（カード追加・修正のたびに確認）
- [ ] カードタイトルに「DSCR」「容積率」「CF」などの略語・専門用語だけを使っていないか
- [ ] カードの冒頭説明が「この計算機では〜」でなく「〜したい人はここで計算できます」形式か
- [ ] 入力欄のラベルが「金利（%）」だけでなく「金利（例: 0.5%）」など参考値付きか
- [ ] 結果の数字の後に「（この数字が高いほど〜）」などの解釈ガイドが付いているか
